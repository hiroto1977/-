## 1. 依頼の背景と目的

Service Hub は、個人事業主〜中小企業向けに 64 サービスを統合する業務ダッシュボードである
（Electron デスクトップ版 + ブラウザ単体 HTML 版の 2 ターゲット・1 コードベース）。
現行システムの機能拡張・保守開発を外部委託するにあたり、最適な提案を募る。

## 2. 委託範囲（スコープ）

### 2.1 対象業務（提案対象）
- [ ] Phase 6 拡張開発（例: live fetcher 実装拡大・UI/UX 改善・新サービス追加）
- [ ] 保守・障害対応（SLA は提案に含めること）
- [ ] テスト・品質保証の補強
- ※ 依頼時に該当項目へチェックし、詳細は別紙の要件定義書・プロジェクト計画書 §WBS を参照。

### 2.2 対象外（発注者側で実施）
- 本番環境の認証情報・トークン管理、外部 SaaS の契約・課金
- AIカウンセラーの**安全クリティカルな検知ルール・相談窓口情報の確定判断**
  （委託先は提案まで。確定は発注者の書面承認による — specs/05 §7）
- リリース判断・ストア/配布チャネルの運用

## 3. 技術的前提条件（必須・変更不可）

| 項目 | 内容 |
|---|---|
| 技術スタック | TypeScript / React 18 / Electron 33 / Vite / Vitest / Stryker / Node 22 |
| アーキテクチャ | 3 プロセス分離・`window.serviceHub` ブリッジ経由のみ（`nodeIntegration` 禁止） |
| 単一の真実源 | サービス定義は `src/shared/serviceId.ts`。追加は scaffold ツール使用 |
| 品質ゲート | typecheck / vitest（5,000+ tests）/ verify:all / Stryker mutation 100% / build:web の全グリーン維持 |
| 開発フロー | フィーチャーブランチ → Pull Request → 発注者レビュー → マージ。main 直 push 禁止 |
| ドキュメント | アーキ変更時は `docs/ARCHITECTURE.md` 同期必須（`verify:arch` が自動検査） |

## 4. 提案書に含めるべき内容

1. **会社概要・類似開発実績**（TypeScript/Electron/React の実績を含む）
2. **体制案**（PM・開発者・QA の人数とスキル。再委託の有無と範囲 — 再委託は事前承認制）
3. **開発プロセス**（ブランチ運用・レビュー・テスト方針が §3 の前提に適合すること）
4. **スケジュール案**（マイルストーン・クリティカルパス）
5. **見積**（[03_ESTIMATE.md](./03_ESTIMATE.md) の様式。内訳: 設計/実装/テスト/PM）
6. **品質保証計画**（品質ゲートの維持方法、Stryker 100% の運用経験があれば記載）
7. **セキュリティ対応**（[06_SECURITY_REQUIREMENTS.md](./06_SECURITY_REQUIREMENTS.md) への適合宣言）
8. **リスクと前提条件**（提案の前提・除外事項を明記）
