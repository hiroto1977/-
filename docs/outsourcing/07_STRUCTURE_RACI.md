## 1. 体制図

```
┌──────────────── 発注者（甲） ────────────────┐   ┌──────────────── 受託者（乙） ────────────────┐
│                                              │   │                                              │
│  プロジェクトオーナー（最終意思決定）          │   │  受託責任者（契約・体制責任）                  │
│        │                                     │   │        │                                     │
│  プロダクト責任者（仕様確定・優先順位）        │◀──▶│  PM（進捗・品質・課題管理）                   │
│        │                                     │   │        │                                     │
│  技術レビュー担当（PR レビュー・マージ権限）   │◀──▶│  開発リーダー ── 開発者 / QA                  │
│  セキュリティ担当（要件書の遵守確認）          │   │  （要員名簿は別途提出・更新時は通知）          │
└──────────────────────────────────────────────┘   └──────────────────────────────────────────────┘
                 連絡経路: GitHub Issue / PR（正式記録） + チャット（日常連絡）
```

## 3. RACI チャート

R=実行責任 / A=説明責任（1つのみ）/ C=協議 / I=報告受領

| 活動 | 甲 オーナー | 甲 プロダクト | 甲 技術レビュー | 乙 PM | 乙 開発 |
|---|---|---|---|---|---|
| 要件・仕様の確定 | A | R | C | C | I |
| 設計・実装 | I | C | C | A | R |
| PR レビュー・マージ | I | I | **A/R** | C | C |
| 品質ゲート維持（CI 全グリーン） | I | I | C | A | R |
| AIカウンセラー安全ルール変更の承認 | **A** | R | C | C | I |
| 仕様変更の承認（[08](./08_CHANGE_MANAGEMENT.md)） | A | R | C | C | I |
| 進捗・課題報告（週次） | I | I | I | A/R | C |
| 検収判定（[09](./09_ACCEPTANCE.md)） | A | R | R | C | I |
| インシデント第一報（[06](./06_SECURITY_REQUIREMENTS.md) §6） | I | I | I | A/R | R |
| リリース判断・公開 | A/R | C | C | I | I |

## 4. 会議体・報告

| 会議体 | 頻度 | 参加 | 内容 |
|---|---|---|---|
| キックオフ | 開始時 | 全員 | 体制・スコープ・開発標準・本書の確定 |
| 週次定例 | 週1 | 甲プロダクト・技術 / 乙 PM・リーダー | 進捗・課題・リスク（乙 PM が議事録、Issue に記録） |
| 検収会 | 納品時 | 甲全員 / 乙 PM | 検収基準書に基づく判定 |

- 正式な合意・決定は **GitHub Issue / PR 上の記録**を正とする（チャットは参考）。
- 連絡応答 SLA: 営業日 1 日以内の一次応答（甲乙双方）。
