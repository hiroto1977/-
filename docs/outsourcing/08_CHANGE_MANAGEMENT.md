## 1. 変更管理（仕様・スコープ・納期・金額の変更）

### 1.1 原則
- スコープ・仕様・納期・金額に影響する変更は、**本手順による書面合意（変更合意書）**が
  なければ効力を生じない。
- 軽微な実装上の判断（仕様の範囲内のリファクタリング等）は PR レビューでの合意で足りる。

### 1.2 手順

```
① 起票        変更要求 (CR) を GitHub Issue で起票（甲乙どちらからでも可）
                 ラベル: change-request / 記載: 背景・変更内容・希望期日
② 影響分析    乙が 5 営業日以内に影響分析を回答
                 (a) 工数・金額への影響（追加見積: 03 の単価表に基づく）
                 (b) 納期への影響  (c) 品質・安全要件への影響
③ 判定        甲プロダクト責任者が 採用 / 不採用 / 保留 を判定（RACI: A=オーナー）
                 ※ AIカウンセラー安全要件に触れる変更は甲オーナーの書面承認が必須
④ 合意        採用の場合「変更合意書」(§1.3) を取り交わし、個別契約の一部とする
⑤ 反映        Issue を仕様の出典として実装 → PR → 検収対象に追加
```

### 1.3 変更合意書（様式）

| 項目 | 内容 |
|---|---|
| CR 番号 / Issue | #＿＿＿ |
| 変更内容 | |
| 委託料の増減 | ¥＿＿＿＿（税抜） |
| 納期の変更 | 変更前: ____ → 変更後: ____ |
| その他条件 | |
| 合意日・記名押印 | 甲: ＿＿＿＿ 印 / 乙: ＿＿＿＿ 印 |

## 2. 課題管理

| 項目 | 運用 |
|---|---|
| 記録場所 | GitHub Issues（単一の課題台帳。チャットでの報告も Issue に転記） |
| 必須項目 | 事象 / 再現手順 / 影響 / 期限 / 担当（RACI に整合） |
| 優先度 | P1: 安全・セキュリティ・データ破損（即日着手） / P2: 機能ブロッカー（3営業日） / P3: 通常（次回定例で計画） |
| 追跡 | 週次定例でオープン課題を棚卸し。期限超過は定例でエスカレーション |

## 3. 障害（不具合）の扱い

- **検収前**: 不具合修正は委託料の範囲内（請負の完成義務）。
- **検収後 1 年以内**: 契約不適合責任（基本契約第8条）として無償修補。
- **仕様変更に起因するもの**: §1 の変更管理による（有償）。
- 切り分けに争いがあるときは、要件定義書・設計書・Issue 記録を正として甲乙協議する。

## 4. エスカレーションパス

```
担当者間で 3 営業日解決しない → 乙 PM ⇄ 甲プロダクト責任者 → 5 営業日解決しない
→ 乙受託責任者 ⇄ 甲プロジェクトオーナー（契約協議）
```
