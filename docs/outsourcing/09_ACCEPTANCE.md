# 検収基準書・検収書様式

| 項目 | 内容 |
|---|---|
| 文書名 | 検収基準書 (Acceptance Criteria) + 検収書様式 |
| 位置づけ | 基本契約第5条・個別契約 §3 の検収はすべて本書による |
| 特徴 | **検収基準の大半が機械検証可能**（CI の品質ゲート）— 主観に依存しない |

---

## 1. 検収の流れ

```
① 納入      乙が PR を提出（ソース + テスト + ドキュメント同梱）
② 自動検査  CI (.github/workflows/ci.yml) が §2 の必須基準を自動判定
③ 受入確認  甲が §3 の受入観点を確認（10 営業日以内）
④ 判定      合格 → 検収書(§5)発行・マージ / 不合格 → §4 の手続
```

## 2. 必須基準（自動・全項目グリーンが合格条件）

| # | 基準 | 検証コマンド | 合格条件 |
|---|---|---|---|
| 1 | 型整合 | `npm run typecheck` | エラー 0 |
| 2 | 単体テスト | `npm test` | 全件パス（既存 5,000+ 件を含む） |
| 3 | アーキ整合 | `npm run verify:arch` | `docs/ARCHITECTURE.md` の file:line 参照・メトリクス一致 |
| 4 | 禁止パターン | `npm run lint:forbidden` | `nodeIntegration` / `innerHTML` / `eval` 等の検出 0 |
| 5 | プロセス境界 | `npm run lint:imports` | 境界違反 0 |
| 6 | 文書一貫性 | `npm run lint:docs` | クロスドキュメント事実の不一致 0 |
| 7 | テスト網羅 | `npm run lint:test-coverage` | 全サービスにテストとアクション登録 |
| 8 | 変異テスト | `npm run mutate` | Stryker mutation score 100%（等価変異は pragma で根拠明示） |
| 9 | ビルド | `npm run build:web` | `dist/standalone.html` が生成され非自明なサイズ |

> 新規コードには新規テストを伴うこと。**既存テストの削除・無効化・閾値の引き下げは
> それ自体を不合格事由とする**（特に AIカウンセラー安全不変条件: 危機→窓口照会 100% /
> safety-miss 0 / 確証済み窓口のみ）。

## 3. 受入観点（甲による確認）

| # | 観点 | 内容 |
|---|---|---|
| 1 | 仕様適合 | 個別契約 §2 のスコープ・参照仕様（specs / Issue）との一致 |
| 2 | 受入テスト | 個別契約に定めた受入シナリオの実施（デスクトップ起動 `npm run dev` / ブラウザ版 `dist/standalone.html`） |
| 3 | UI/文言 | 日本語表記・トーンの統一、免責文言の維持 |
| 4 | ドキュメント | 変更に応じた `docs/` の更新（ARCHITECTURE / 該当 specs） |
| 5 | セキュリティ | [06_SECURITY_REQUIREMENTS.md](./06_SECURITY_REQUIREMENTS.md) §3〜4 の遵守（依存追加の事前承認記録を含む） |

## 4. 不合格時の手続

1. 甲は不合格事由を **GitHub PR レビューコメントまたは Issue** で具体的に指摘する。
2. 乙は甲乙協議のうえ定めた期限までに無償で修補し、再納入する（§1 を再実施）。
3. 同一事由で 3 回不合格となった場合は、甲乙責任者間で対応を協議する
   （[08_CHANGE_MANAGEMENT.md](./08_CHANGE_MANAGEMENT.md) §4 のエスカレーション）。
