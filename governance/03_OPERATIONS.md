# 03. 日常運用ルールとインシデント対応

## A. 業務開始前 (毎朝)

```
□ scripts/preflight.sh を実行 → 全項目が ✅
□ scripts/storage-orchestrator.sh --routine daily (健康診断のみ)
□ scripts/funding-deadline.sh (補助金/融資期限の警告)
□ Ollama サーバー起動確認 (curl http://localhost:11434/api/tags)
□ 暗号化ボリュームのマウント確認
□ ネットワーク (社内 / 公衆 / VPN) を確認
□ 今日扱う案件の最高データクラスを把握 (C0〜C4)
```

## B. 入力時のルール (AI に何かを投げる前)

### B-1. 必ず行う 3 ステップ

1. **分類**: 入れようとしている情報のデータクラス (`02_DATA_CLASSIFICATION.md` 参照)
2. **道具選択**: マトリクスに従いツール選定
3. **マスキング**: クラウドへ送るならテンプレ (`05_TEMPLATES.md`) でマスキング

### B-2. 入力前チェックリスト

```
□ このデータは C0/C1/C2 か (C3 以上ならローカル必須)
□ 個人名・社名・電話番号・メール・住所・口座番号・マイナンバー・健康情報を含まないか
□ NDA 対象データを含まないか
□ 第三者著作物を不当に再利用しようとしていないか
□ クラウドに送る場合: 学習 opt-out 済みの API か
□ 出力を社外公開する予定なら、根拠資料の利用可否を確認したか
```

## C. 出力時のルール (AI が返したものを使う前)

### C-1. レビュー必須項目

| 項目 | 確認 |
|---|---|
| **事実誤認** | 数値・固有名詞・引用元 |
| **ハルシネーション** | 存在しない法律名・判例・URL を検証 |
| **個人情報の混入** | 元データから漏れた識別子 |
| **既存著作物との類似** | 文章は剽窃チェックツールで |
| **差別的・攻撃的表現** | 偏見を反映していないか |
| **根拠の弱い断定** | 「必ず」「絶対」等の言い切りを再検証 |

### C-2. 二次利用前のレビュー

AI 出力をそのまま社外提出 (顧客提案、契約書、申告書類) するのは禁止。**必ず人間レビュー → 修正 → 承認** を経る。

## D. ファイル管理

### D-1. ファイル命名規則

```
[クラス]_[日付]_[案件]_[内容].拡張子
例: C3_20260504_顧客A_提案書.docx
   C2_20260504_見積案_α社.xlsx
   C0_20260504_press_release.md
```

### D-2. フォルダ構成例

```
~/work/
├── C0_public/         # 公開可
├── C1_internal/       # 社内一般
├── C2_partner/        # 取引先
├── C3_confidential/   # 機密 (要暗号化)
└── C4_legal/          # 法定機微 (専用領域)
```

`C3_*` 以上は **VeraCrypt / FileVault / dm-crypt 等の暗号化ボリューム**配下に置く。

### D-3. バックアップ

| クラス | バックアップ方針 |
|---|---|
| C0/C1 | 通常クラウドストレージで可 |
| C2 | 暗号化済みクラウドストレージ + 別経路の鍵管理 |
| C3 | ローカル外付け SSD (暗号化) ＋ 週次オフサイト保管 (耐火金庫等) |
| C4 | ローカル + 物理的別場所のオフライン媒体のみ |

### D-4. 監査ログ (`~/.claude/audit.jsonl`) の オフライン バックアップ

監査ログ全体を同一端末で管理する以上、**端末紛失/破損で全消失する**。
月次の自動バックアップ (`~/.claude/audit-backups/audit.jsonl.bak.YYYYMM`) も同じ端末上にあるため、根本対策にはならない。

**運用フロー** (月次実施を推奨):

```sh
# 1. エクスポート (現行 audit.jsonl + 全ての .bak を一つの tar.gz に)
bash scripts/audit-export.sh ~/Desktop/audit-export
# → ~/Desktop/audit-export/audit-export-YYYYMMDDHHMMSS.tar.gz
# → ~/Desktop/audit-export/audit-export-YYYYMMDDHHMMSS.manifest.json (sha256)

# 2. USB / 外付け SSD / 別端末 にコピー
cp ~/Desktop/audit-export/audit-export-*.tar.gz /Volumes/USB-BACKUP/

# 3. アンマウントして物理分離
sync && diskutil eject /Volumes/USB-BACKUP   # macOS
# または: udisksctl unmount -b /dev/sdXN     # Linux

# 4. 取り込み 検証 (別端末 / 復旧時)
tar -xzf audit-export-*.tar.gz
bash scripts/audit-verify.sh audit-export/audit.jsonl
```

**運用ルール**:
1. **頻度**: 最低 月次。重要な業務節目 (案件完了 / 法定報告前) の都度推奨
2. **媒体ローテーション**: USB/SSD は 2 本以上を交互利用 (片方故障時の保険)
3. **保管場所**: バックアップ媒体は **元端末と物理的に別の場所** に置く (耐火金庫 / 別オフィス / 信頼できる第三者預かり)
4. **C4 の取り扱い**: 監査ログが C4 データの存在を示唆する場合、バックアップ媒体自体も C4 として扱う (暗号化必須)
5. **検証**: 新しいエクスポートを取った後、**過去のエクスポートを 1 つ展開して `audit-verify.sh` が通ることを確認** (媒体劣化の早期発見)

**廃棄時**: 古い USB を廃棄する際は **物理破壊** (ハンマー / 専用シュレッダ)。論理削除では不十分。

## E. アクセス制御

| 役割 | 権限 |
|---|---|
| 経営者・法定代理人 | 全クラス参照可 |
| 主任担当 | 担当案件のみ C0〜C3 |
| 一般メンバー | C0〜C2 全般、C3 は許可制 |
| パート・業務委託 | C0〜C1 のみ (契約により C2 拡張) |
| 退職者 | 即時アクセス停止 (チェックリスト要) |

## F. ログと監査

### F-1. 残すべき最低限のログ

- AI 利用記録: いつ、誰が、どのツール (cloud/ollama)、どのモデル、どのデータクラス
- ファイルアクセス: C3 以上のファイルへのアクセスログ (OS の監査機能 or 専用ツール)
- 退職時のデータ削除証跡

### F-2. 簡易ログのつけ方

```bash
# AI 利用ログを 1 行 1 リクエストで記録 (例)
echo "$(date -Iseconds) [user=$USER] [tool=ollama] [model=llama3.2] [class=C2] [purpose=要約]" >> ~/ai-usage.log
```

監査時に検索可能な形 (CSV / JSON Lines) で残す。

## G. インシデント対応 (発見〜事後)

### G-1. インシデントの定義

- 個人情報・営業秘密の意図しない外部送信
- AI 出力に他人の機密が混入していたことが事後判明
- アカウント乗っ取り疑義
- バックアップ媒体の紛失

### G-2. 即時対応 (発見〜2 時間)

```
1. □ 拡散の停止 (関連 API キー失効、共有リンク削除、ファイル隔離)
2. □ 証拠の保全 (該当ログ・スクリーンショット・関連ファイルの copy)
3. □ 関係者へ第一報 (管理者・法務担当・契約上の責任者)
4. □ 影響範囲の暫定特定 (誰のどのデータが、どこへ)
```

### G-3. 短期対応 (24〜72 時間)

```
5. □ 個人情報保護委員会への速報 (要配慮 or 1000 人超 or 不正アクセス起因 or 財産的被害)
   速報期限: 漏えい等を知った日から概ね 3〜5 日 (個人情報保護法施行規則 8 条)
6. □ 本人通知 (該当する場合)
7. □ 取引先・委託元への通知 (NDA/契約上の義務確認)
8. □ ベンダ (AI 事業者) のサポートに連絡 (データ削除依頼)
```

### G-4. 中期対応 (1 週間〜30 日)

```
9. □ 確報を個人情報保護委員会へ (期限 30 日。不正アクセス起因は 60 日)
10. □ 根本原因分析 (Why × 5 回)
11. □ 再発防止策の策定と実装
12. □ 関係者への報告 (社内・取引先・必要に応じプレス)
```

### G-5. 連絡先テンプレ

```
[宛先]: 個人情報保護委員会
[件名]: 個人データ漏えい等の発生に係る速報
[本文]:
1. 概要 (発生日時・覚知経緯)
2. 漏えい等した個人データの項目と件数
3. 原因
4. 二次被害の有無
5. 本人への対応状況
6. 公表予定
7. 担当者連絡先
```

詳細様式は個人情報保護委員会の公式雛形を使用。

| 周期 | コマンド | 内容 |
|---|---|---|
| **毎日** | `bash scripts/storage-orchestrator.sh --routine daily` | health のみ (1 分以内) |
| **毎週** | `bash scripts/storage-orchestrator.sh --routine weekly` | health → cleanup --dry-run (5 分) |

**cron 連携例** (毎週月曜 9:00 に weekly):

```sh
0 9 * * 1  cd /path/to/repo && bash scripts/storage-orchestrator.sh --routine weekly --json-report > /tmp/storage-weekly.json
```
