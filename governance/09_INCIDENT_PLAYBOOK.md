# 09. インシデント レスポンス プレイブック

事象が発覚した時に「何を何分以内に行うか」を、よくある 8 シナリオで具体化。
普段の `governance/03_OPERATIONS.md` G 節と併読。

> ⚠️ 本書は緊急時の即応用。**法的判断・大型インシデント** は弁護士・専門家に即時相談すること。

## 共通: 最初の 60 秒で打つコマンド

```sh
# 1. 現状を凍結 (調査用スナップショット)
date -Iseconds > /tmp/incident-$(date +%Y%m%d-%H%M%S).log
hostname >> /tmp/incident-*.log
whoami >> /tmp/incident-*.log
git -C ~/path/to/repo log --oneline -20 >> /tmp/incident-*.log 2>&1 || true
last -n 20 >> /tmp/incident-*.log 2>&1 || true

# 2. ネットワーク状態を凍結
ss -tnp 2>/dev/null > /tmp/incident-net.txt || netstat -an > /tmp/incident-net.txt

# 3. PII スキャン (該当する場合)
bash scripts/pii-scan.sh --diff > /tmp/incident-pii.txt 2>&1 || true

# 4. 現場の関係者に第一報 (LINE/Signal 等の独立チャネル)
```

これらの **証跡保全 → 関係者へ通知 → 事態評価** の順は全シナリオ共通。

## I-1. シークレット (API キー / .env) を git にコミット

**`governance/08_ATTACK_CATALOG.md` D-2 と対応**

### 検知シグナル
- `bash scripts/pii-scan.sh --diff` で API キーパターン検出
- GitHub の secret scanning アラート
- 第三者からの「あなたのキーで怪しい挙動が」連絡

### 60 秒で打つコマンド
```sh
# 1. キー失効が最優先 (履歴削除より先)
#    Anthropic Console / Google AI Studio / その他 でキーを Revoke
#    新しいキーを発行

# 2. 証跡保全
git log --all --full-history -p -- .env > /tmp/secret-history.log

# 3. 公開済か確認
git remote -v
gh repo view --json visibility 2>/dev/null
```

### 評価 (10 分)
- 公開リポジトリか / Private か
- 公開期間 (push から発覚まで何時間か)
- 影響キーの権限 (read-only / read-write / admin)
- 当該キーで実行されたアクション ログ (各 API のダッシュボード)

### 封じ込め (30 分)
1. ✅ キー失効を確認 (新キーで認証成功 + 旧キーで認証失敗)
2. 履歴からキーを削除 (`git filter-repo` 等) — **公開鍵管理視点では失効が先**
3. 強制 push 必要なら **チームに事前通知**
4. 関係するクラウド アカウントの不審アクセス確認

### 復旧
- 業務再開: 新キーで動作確認
- アクセス ログを 7 日間 通常より厳しく監視

### 学び
- pre-commit hook 強化 (`scripts/pii-scan.sh --staged`)
- `.gitignore` に `.env*` 追加
- ファイル名規則の見直し

## I-2. クラウド AI に機密データを誤送信

**`governance/08_ATTACK_CATALOG.md` H-1, J-1 と対応**

### 検知シグナル
- 送信直後の社員からの自己申告
- AI 履歴のレビューで C3 以上のキーワード発見
- 取引先からの「弊社情報がそちらの AI 利用ログに?」問い合わせ

### 60 秒で打つコマンド
```sh
# 1. AI ベンダの「データ保持期間」を即時確認
#    Anthropic: 30 日 / Zero Retention 申請可
#    OpenAI: 30 日 / Zero Retention 申請可
#    Google: 構成による

# 2. 該当チャット セッションの即時削除
#    各サービスの会話履歴を手動削除 (証跡として screenshot を先に取る)

# 3. 当該データの個人特定可能性を判定
```

### 評価 (1 時間)
- 投入されたデータの **データ クラス** (`governance/02_DATA_CLASSIFICATION.md`)
- 含まれる本人の数 (1 / N / 1000+)
- 要配慮個人情報 / マイナンバーの含有
- 学習目的での利用 opt-out 設定の有無

### 封じ込め (24 時間)
1. AI ベンダのサポートに連絡 → データ削除依頼書面 (DPA に基づく)
2. 法務確認: 第三者提供 (法 27 条) の例外に該当するか
3. 1000 人超 / 要配慮 / 不正アクセス由来 → **個情委への速報** (3〜5 日以内)
4. 該当する本人通知の準備

### 復旧
- ローカル専用モード (`v19/`) を全社規定化
- AI 入力前 PII マスキングの徹底 (`governance/05_TEMPLATES.md` A 節)

### 学び
- 入力前チェックリストの簡素化
- マスキング ツールの自動化
- 再発防止訓練 (該当者 + 全社共有)

## I-3. ノート PC 紛失 (C3 以上のデータ)

**`governance/08_ATTACK_CATALOG.md` G-1 と対応**

### 検知シグナル
- 社員からの自己申告
- 紛失届
- 端末からの heartbeat 喪失 (MDM 利用時)

### 60 秒で打つコマンド
```sh
# MDM (Jamf / Intune / Kandji 等) からリモート ロック / ワイプ
#  例: gam user [user] remote lock      (Google Workspace)
#       hibp lock [device]               (Microsoft 365)
```

### 評価 (1 時間)
- ディスク暗号化の有無 (FileVault / BitLocker / LUKS)
- パスワード強度 / 自動ロック時間
- 直近のクラウド同期 (オフライン編集物の有無)
- 業務用アプリの ログイン状態 (Cookie 残存)
- 含有 データクラス (C0-C4)

### 封じ込め (24 時間)
1. 全クラウド アカウントの当該端末の セッション失効
2. 全 SaaS のパスワード ローテーション (該当者)
3. 警察 (盗難届) / 鉄道・タクシー会社 (遺失届)
4. 紛失先住所が特定できれば現地確認
5. 該当する補助金 / 助成金 申請 関連書類があれば事務局へ報告

### 復旧
- 代替端末の準備
- データ復旧 (バックアップ / クラウドから)

### 学び
- 全社員のディスク暗号化 ON 確認
- 自動ロック時間 5 分以内 強制
- 業務アプリの 30 日 ログイン期限

## I-4. ランサムウェア感染兆候

**`governance/08_ATTACK_CATALOG.md` I-1 と対応**

### 検知シグナル
- ファイル拡張子の異常変化 (`.encrypted`, `.locked` 等)
- デスクトップに「身代金要求」テキスト
- 共有 ドライブの全ファイル更新時刻 同時変動
- アンチウイルスのアラート

### 60 秒で打つコマンド
```sh
# 1. ネットワーク 即時遮断 (Wi-Fi / 有線)
#    "本番環境への被害拡大" を防ぐ

# 2. シャットダウンしない (メモリ上の証跡保全)
#    可能なら hibernate (RAM 内容をディスクに保存)

# 3. 他端末・サーバーへの拡散を確認
ping all-internal-hosts.txt 2>&1 > /tmp/ransom-spread.log
```

### 評価 (即時)
- 影響台数 (1 台 / 複数 / 全社)
- 影響データのクラス (C0-C4)
- バックアップの最終時刻 (RPO)
- 暗号化 アルゴリズム / 既知の復号ツール存在

### 封じ込め (24 時間)
1. 影響端末の物理隔離 (LAN ケーブル抜く)
2. 警察 (サイバー犯罪相談窓口)
3. **身代金は払わない** (再発・追加要求リスク)
4. バックアップからの復旧開始 (隔離環境で)
5. 弁護士 即時相談 (社外公表の判断)

### 復旧
- 完全初期化 + クリーン OS インストール
- バックアップから データ復旧
- パスワード全交換

### 学び
- 3-2-1 バックアップ ルールの強化
- フィッシング メール訓練
- EDR (Endpoint Detection & Response) 導入検討

## I-5. フィッシング被害 (パスワード入力済)

**`governance/08_ATTACK_CATALOG.md` A-1 と対応**

### 検知シグナル
- 自己申告 (「あの URL に入力しちゃった」)
- 「不審なログイン試行」通知メール
- アカウントから自分が出していない投稿 / メール

### 60 秒で打つコマンド
```sh
# 1. 該当アカウントのパスワードを即変更
#    + 全アクティブ セッション失効 (各サービスの "全デバイスからログアウト")

# 2. MFA を即時有効化 (まだなら)

# 3. 同じパスワードを使っている他サービスを思い出す → 全変更
```

### 評価 (30 分)
- 漏れたアカウントから アクセス可能だった データのクラス
- 連動サービス (Google / Apple ID / Microsoft) があるか
- 既に攻撃者が何かしているか (送信箱・履歴を確認)

### 封じ込め (4 時間)
1. 連動サービスの全パスワード変更
2. メールの転送ルール / フィルタを精査 (攻撃者が転送を仕掛ける常套手段)
3. 信頼するデバイス / アプリ から見覚えないものを除去

### 復旧
- パスワード マネージャの導入 (再利用回避)
- 全 SaaS の MFA 必須化

### 学び
- フィッシング訓練 (年 2 回)
- URL クリック前の hover 確認の徹底

## I-6. Ollama ポート (11434) が外部公開された疑い

**`governance/08_ATTACK_CATALOG.md` (新興 / 本リポジトリ固有)**

### 検知シグナル
- ルーター / ファイアウォールの 11434 ポート ログ
- Shodan / Censys 等での発見通知
- 知らない IP からの API 呼び出し履歴

### 60 秒で打つコマンド
```sh
# 1. Ollama 即停止
pkill -f "ollama serve"

# 2. ファイアウォールの 11434 ポートを閉じる
sudo ufw deny 11434         # Linux ufw
# または ルーターのポート転送を削除

# 3. OLLAMA_HOST を localhost のみに固定
echo "OLLAMA_HOST=127.0.0.1:11434" >> ~/.bashrc

# 4. アクセス ログ調査 (もしあれば)
journalctl -u ollama -n 1000 > /tmp/ollama-access.log 2>/dev/null
```

### 評価
- 公開期間 (いつから?)
- アクセス元 IP の地理位置 / 既知の攻撃者リスト
- 利用したモデル + 引き出された応答内容の推定

### 封じ込め
1. localhost バインドの強制
2. もし VPN 経由で意図的公開だった場合は VPN 設定見直し
3. 業務 PC 以外 (ホームサーバー等) で動かしていた場合の用途見直し

### 復旧 / 学び
- Ollama 起動時 必ず `OLLAMA_HOST=127.0.0.1` 確認
- スマートホーム機器との同居を避ける (UPnP で意図せずポート開放)

## I-7. 退職者による顧客データ持ち出し疑い

**`governance/08_ATTACK_CATALOG.md` I-3 と対応**

### 検知シグナル
- 退職予告から退職日までの異常な C3 アクセス急増
- USB / プリンタ ジョブ ログの異常
- 退職後に競合他社が同じ顧客リストでアプローチ
- 元同僚からのリーク

### 60 秒で打つコマンド
```sh
# 1. 全アクセス権の即時剥奪 (退職プロセスをやり直し)
# 2. 当該人物のアクセス ログ過去 90 日分 を取得・凍結
# 3. データ ダウンロード履歴 / メール送信履歴 / 印刷履歴
```

### 評価
- 在職時の秘密保持誓約書の有無
- 営業秘密の 3 要件適合 (`governance/01_LEGAL_FRAMEWORK.md` C 節)
- 民事 (損害賠償・差止) / 刑事 (不正競争防止法 違反) の選択

### 封じ込め
1. 弁護士相談 → 内容証明郵便 (退職者へ警告)
2. 競合他社へ警告通知 (情報利用は不正競争防止法違反)
3. 関係顧客への状況説明 (信頼維持)

### 学び
- 退職プロセスの強化 (`governance/06_ONBOARDING.md` F 節)
- 在職時の秘密保持誓約書の存続期間条項
- C3 へのアクセス監査ログ常設

## I-8. SNS で機密情報の漏洩を発見

**`governance/08_ATTACK_CATALOG.md` D-3, F-1 と対応 / `templates/crisis/sns_crisis.md` も併用**

### 検知シグナル
- Google アラート / SNS 監視サービスの通知
- 顧客 / 同業者からの連絡
- 第三者の SNS 投稿のスクリーンショット

### 60 秒で打つコマンド
```sh
# 1. 投稿の証跡保全 (スクリーンショット + Wayback Machine)
curl "https://web.archive.org/save/[URL]"

# 2. 投稿者を特定 (社内 / 社外)

# 3. 公開削除依頼の準備 (社内なら本人へ、社外なら法的措置検討)
```

### 評価
- 投稿の機密度 (C0-C4 のどれが漏れたか)
- 拡散範囲 (リポスト数 / インプレッション)
- 原因者 (社員 / 退職者 / 第三者 / 内部関係者)

### 封じ込め
1. 社内発信なら **即削除依頼 + 投稿者ヒアリング**
2. 社外発信なら **プロバイダ責任制限法 に基づく削除請求**
3. 投稿者特定が必要なら **発信者情報開示請求** (弁護士)
4. 個人情報含有なら 個情委 報告検討

### 復旧 / 学び
- SNS 運用ガイドライン整備 (`templates/crisis/sns_crisis.md` F 節)
- 私物アカウントでの会社言及制限
- 危機対応訓練

## M. 共通 学習サイクル

```
インシデント発生
    ↓
60 秒対応 (本書)
    ↓
24 時間以内 評価・封じ込め
    ↓
30 日以内 復旧 + 個情委 確報 (該当時)
    ↓
振り返り (Why × 5、根本原因)
    ↓
governance/ ・ templates/ の更新
    ↓
攻撃カタログ (08) のシナリオ追加
    ↓
次のインシデント発生時の対応速度向上
```

毎四半期 末に **過去のインシデント / ニアミス を振り返り**、本書を改訂。
