# 10. ストレージ衛生 (Storage Hygiene)

「Claude Code が動く端末を **常に軽く・速く・安全** に保つ」ための仕組み。

## 0. 基本方針 — 一気にクラウドへ移さない

### 0-1. なぜか

- `02_DATA_CLASSIFICATION.md` のマトリクスにより、**C3 以上のデータは原則ローカル**
- C4 (マイナンバー / 要配慮個人情報) は **クラウド禁止**
- 「とにかく軽くしたい」が動機でも、**法令違反 (個人情報保護法 28 条 越境移転) のリスク** を超えてまで急ぐ価値はない

### 0-2. やる順序

```
1. 把握  : 何がどこに、どのクラスで、どれだけあるか (storage-health.sh)
2. 分類  : C0-C4 で仕分け
3. 削除  : 二度と要らないものは削除 (キャッシュ / 一時 / 重複)
4. 圧縮  : 古いログ・アーカイブを zstd 圧縮
5. 階層化: クラウドへ送れるものだけ送る (storage-archive.sh)
6. 監視  : 月次で衛生指標を確認 + ルーティン
```

「速くする」だけが目的なら、3 → 4 まででほぼ達成できる。クラウド化は最後の手段。

## 1. パフォーマンス目標

| 指標 | 警告 | 目標 | 理想 |
|---|---|---|---|
| **ストレージ空き容量** | < 20% | ≥ 30% | ≥ 50% |
| **メモリ空き** (常時) | < 1 GB | ≥ 2 GB | ≥ 4 GB |
| **inode 残数** (Linux) | < 5% | ≥ 20% | ≥ 50% |
| **swap 使用** | > 50% | < 20% | < 5% |
| **`/tmp` サイズ** | > 5 GB | < 1 GB | < 500 MB |
| **`~/Downloads` 古さ** | 90+ 日 | 30 日以内 | 7 日以内 |
| **ブラウザ キャッシュ** | > 5 GB | < 2 GB | < 1 GB |
| **未コミット変更** | > 100 ファイル | < 20 | 0 |

`scripts/storage-health.sh` で自動測定。

## 2. データ ライフサイクル (5 ステージ)

```
[Hot]                [Warm]              [Cold]              [Frozen]              [Deleted]
作業中            最近触ってる          月単位アクセス        年単位アクセス
SSD ローカル      SSD ローカル          外付け / クラウド     クラウド コールド     永久消去
高速 IOPS         普通                  低頻度                アーカイブ層
~7 日             ~30 日                ~6 ヶ月               長期                  保持期間後
```

判断基準:
- **最終アクセス** (atime) > 30 日 → Warm へ
- > 6 ヶ月 → Cold へ (圧縮 + 暗号化済 アーカイブ層)
- > 2 年 → Frozen へ (アーカイブ層、復元時間 数時間 可)
- 法定保管期間 経過 → 削除 (税務 7 年、労務 3〜5 年、個人情報 利用目的達成後)

## 3. クラス別 保存先 マトリクス

| データ クラス | Hot 配置 | アーカイブ配置 | 暗号化 | バックアップ 3-2-1 |
|---|---|---|---|---|
| **C0** (公開) | 任意 | 任意のクラウド | 不要 | 1 コピーで可 |
| **C1** (社内一般) | ローカル SSD | クラウド ストレージ | 推奨 | 3-2-1 |
| **C2** (取引先) | ローカル SSD | 暗号化クラウド + DPA | **必須** (TLS + AES-256) | 3-2-1 必須 |
| **C3** (機密) | ローカル SSD (暗号化ボリューム) | **クライアント側暗号化** + クラウド or 外付け SSD | **必須** (rclone crypt 等) | 3-2-1 必須 |
| **C4** (法定機微) | ローカル SSD (暗号化) | **クラウド原則禁止**、外付け SSD のみ + 物理金庫 | **必須** | 媒体を地理的に分散保管 |

## 4. 推奨クラウド ストレージ

### 4-1. 評価項目

詳細は `04_VENDOR_REVIEW.md`。要点:

- 学習目的での利用 ❌
- DPA / SCC 締結可
- 第三者認証 (SOC2 II / ISO 27001)
- 越境移転先の明示
- 顧客管理鍵 (CMK) サポート
- バージョン履歴 (ランサムウェア対策)

### 4-2. クラス別 推奨

| クラス | 候補 (五十音順) | 必須条件 |
|---|---|---|
| C0/C1 | Box / Dropbox Business / Google Workspace / iCloud+ / OneDrive Business | — |
| C2 | 上記 + DPA 締結済 | 法人契約 |
| C3 | 上記 + **`rclone crypt`** または **Cryptomator** で クライアント側暗号化 | クライアント鍵を分離保管 |
| C4 | クラウドは使わない | 外付け SSD (FIPS 140-2 認証品) + 物理金庫 |

> **クライアント側暗号化が肝**: クラウド事業者が万一鍵を要求されても (法的・技術的に) 中身を渡せない状態を作る。`rclone crypt` または Cryptomator は実績あり。

### 4-3. 「いいクラウド」の最低ライン

```
□ TLS 1.2 以上 で送信
□ AES-256 で保存時暗号化 (事業者管理 + クライアント側 二重)
□ 多要素認証 必須化可
□ アクセス ログ エクスポート可
□ バージョン履歴 30 日以上
□ 共有リンク 期限・パスワード設定可
□ 監査 (SOC2 / ISO27001) 取得済
□ 日本リージョン or 同等水準国
□ 削除時の物理消去 ポリシー明示
```

## 5. ルーティン

### 5-1. 毎日 (1 〜 5 分)

```
□ scripts/preflight.sh (既存・改修済) で:
  - 空き容量・メモリ・inode が警告以下でないか
□ 当日 終了時:
  - ブラウザのダウンロードフォルダを整理
  - クラウド AI 履歴で機密データ含むセッションは削除
  - 機密ファイルを暗号化ボリューム配下へ
```

### 5-2. 毎週 (15 分)

```
□ scripts/storage-health.sh で詳細チェック
□ scripts/storage-cleanup.sh --dry-run で安全削除 候補確認
□ ブラウザ キャッシュ クリア (週 1)
□ Git の未コミット変更を 0 に近づける
□ 取得済 Ollama モデルの棚卸し (使ってない大型は ollama rm)
□ `~/Downloads` の 30+ 日 経過分を仕分け
```

### 5-3. 毎月 (1 時間)

```
□ scripts/storage-archive.sh で コールド データを アーカイブ
□ クラウド ストレージの容量と請求確認
□ 保管期間経過した法定文書の削除判断 (税理士確認)
□ バックアップ 3-2-1 の物理確認 (リストア テスト 推奨)
□ 「触っていない」ファイル 発見・処理:
   find ~ -atime +180 -size +100M  (Linux/Mac)
□ 重複ファイル スキャン (rdfind / fdupes)
```

### 5-4. 四半期 (半日)

```
□ クラウド アカウント の棚卸し:
  - 不要アカウント解約
  - パスワード ローテーション
  - 2FA 機器の確認
□ rclone の暗号化鍵バックアップ確認
□ 外付け媒体の SMART チェック (smartctl -a)
□ 保管期間ルール (governance/03_OPERATIONS.md F 節) との照合
```

### 5-5. 年次 (1 日)

```
□ 全データ クラス棚卸し
□ クラウド ベンダ評価 (04_VENDOR_REVIEW.md) 再実施
□ ディスク 全体スキャン (chkdsk / fsck)
□ OS / アプリの大型更新 → 古いキャッシュ削除
□ 暗号化方式 の妥当性 (アルゴリズム陳腐化) 再評価
□ アーカイブ媒体の入替 (5+ 年経過 SSD/HDD)
```

## 6. 削除して良いもの / 慎重なもの

### 6-1. 即削除可能 (`storage-cleanup.sh` の対象)

```
✅ ~/.cache/* (Linux/Mac の汎用キャッシュ)
✅ /tmp/* (再起動でも消えるが手動でも可)
✅ ~/Library/Caches/* (Mac)
✅ %TEMP%\* (Windows)
✅ パッケージ マネージャ キャッシュ:
   apt: sudo apt-get clean
   brew: brew cleanup --prune=all
   npm: npm cache clean --force
   pip: pip cache purge
   yarn: yarn cache clean
✅ Docker:
   docker system prune -af --volumes  (※ 重要なボリュームに注意)
✅ ブラウザ キャッシュ (履歴・cookie は残す選択可)
✅ ダウンロード フォルダの DMG / インストーラ (使用済)
✅ 古いログ ファイル (~/.npm/_logs, /var/log/*, ~/.local/share/*/logs/)
```

### 6-2. **削除前に確認** (慎重)

```
⚠️ ~/.ssh/ — 鍵をうっかり消すと復旧不能
⚠️ ~/Documents/* — クラス分類して退避
⚠️ ~/Desktop/ — 一時的な置き場、中身は元あった場所が分からないと損失
⚠️ Ollama モデル (~/.ollama/models/) — 再 pull に時間
⚠️ git リポジトリ — 未 push のローカル ブランチ
⚠️ クラウド同期フォルダ — ローカル削除がクラウドにも伝播する
⚠️ Time Machine / Backups バックアップ
⚠️ 仮想マシン (.vdi / .vmdk / .qcow2)
⚠️ メール ローカル コピー (Outlook .pst, Apple Mail Mailbox)
⚠️ Photos / Videos ライブラリ
```

### 6-3. **絶対に消してはいけないもの**

```
❌ governance/ ・ funding/ ・ templates/ (本リポジトリ)
❌ パスワード マネージャの ローカル DB
❌ 法定保管期間内の経理書類 (税務 7 年)
❌ 未締結契約の進行中ドラフト
❌ 顧客との取引履歴 (別途保管期間)
❌ アクセス ログ (governance/03 F 節)
❌ AI 利用記録 (誰がいつ何を入れたか)
```

## 7. パフォーマンス を悪化させがちな「あるある」

| パターン | 原因 | 解 |
|---|---|---|
| **デスクトップ に 100+ ファイル** | アイコン描画 で起動遅延 | `~/Desktop/inbox/` に集約 |
| **巨大な .git/objects** | ブランチ大量 / 大型バイナリ commit | `git gc --aggressive` / `.git filter-repo` |
| **node_modules 化石化** | 各プロジェクトが冗長 | `npx -y` 主体 / 月 1 で `rm -rf node_modules` |
| **Spotlight / Windows Search index 暴走** | 巨大ファイル を index 中 | プロジェクト dir を index 除外 |
| **Dropbox/Google Drive 同期常時 100%** | 大型ファイル の往復 | 選択同期 / Smart Sync オン |
| **古い Docker コンテナ・イメージ** | レイヤ蓄積 | `docker system prune -af` |
| **Claude Code セッション履歴** | `~/.claude/projects/` 肥大化 | 古いセッション ファイル の選別アーカイブ |
| **Ollama モデル 大量** | お試し pull が貯まる | `ollama list` → 使わないものを `ollama rm` |
| **swap が常用** | 物理メモリ不足 / メモリ リーク | アプリ再起動 / 不要常駐削減 / RAM 増設 |

## 8. クラウド 移行 の安全ワークフロー (rclone crypt 経由)

C2/C3 データを暗号化 + クラウド アーカイブする標準パス。

### 8-1. 一度だけ: rclone セットアップ

```sh
# 1. インストール
curl https://rclone.org/install.sh | sudo bash

# 2. 暗号化前のリモート (例: Box)
rclone config
# → name: box-raw
# → type: box (or s3 / drive / etc)
# → 認証

# 3. 暗号化レイヤ
rclone config
# → name: box-crypt
# → type: crypt
# → remote: box-raw:archive-encrypted
# → password: (ランダム生成、パスワードマネージャに保存)
# → password2 (salt): (ランダム生成、別場所に保存)
```

> 鍵 (password / salt) を失うと **二度と復号できない**。
> パスワード マネージャ + 紙のオフライン バックアップ で 二重化必須。

### 8-2. 通常運用

```sh
# 機密度 C2-C3 のフォルダを暗号化アップロード
rclone copy ~/work/C3_confidential/old_archive box-crypt:2026/

# 確認 (中身は読めるが クラウド管理画面では見えない)
rclone ls box-crypt:2026/

# 必要なら復号ダウンロード
rclone copy box-crypt:2026/ ~/restore/
```

### 8-3. アクセス制御

```
□ rclone 設定ファイル ~/.config/rclone/rclone.conf に password を保存しないモード推奨
□ --password-command で都度入力 or 1Password CLI 連携
□ クラウド 側のアクセス権を最小化 (このユーザーのみ)
```

## 10. 仕組化のための実行可能ファイル

| ファイル | 役割 |
|---|---|
| `scripts/storage-health.sh` | 現状の健康診断 (空き容量・大型ファイル・キャッシュ) |
| `scripts/storage-cleanup.sh` | 安全削除 (キャッシュ / 一時) — `--dry-run` 既定 |
| `scripts/storage-archive.sh` | rclone 経由でアーカイブ層へ移動 (クラス別) |
| `scripts/preflight.sh` (既存) | 業務開始前 — 警告閾値到達なら作業前に対処を促す |

これら 4 つを組み合わせれば、**意図せずに肥大化** することは構造的に起きない。

Ollama モデル (~/.ollama/models/) は 1 モデルあたり 4〜40 GB。

```
□ 用途別に 必要最小モデル 数 (text 用 1 + コード用 1 + vision 用 1 = 計 3 程度)
□ 試したけど常用しないモデル は 即 ollama rm
□ 月 1 で ollama list → 過去 30 日 未使用は削除候補
□ チームで共有する場合は別途 NAS or 共有マウント に集約
```

## 12. NG パターン (やりがちで効果が薄い)

| NG | なぜダメか | 代替 |
|---|---|---|
| 「とりあえず全部 Dropbox」 | C3 以上の漏洩リスク + 月額費用増大 | クラス別 マトリクス (§3) |
| 「定期的に SSD 全消去・OS 再インストール」 | 履歴消失 + 復元コスト | 段階的 cleanup |
| 「暗号化なしで USB 持ち出し」 | 紛失 = 重大漏洩 | 暗号化 USB or rclone crypt 経由 |
| 「圧縮で全部 zip」 | 検索性消失・破損リスク | 古いものは tar.zst、現役は無圧縮 |
| 「CCleaner 等の `クリーンアップ` ソフト過信」 | レジストリ破壊 / マルウェア混入歴 | OS 標準 + 自分の script |
| 「クラウド = バックアップと誤認」 | 同期は バックアップではない (誤削除が伝播) | バージョン履歴 ありの真のバックアップ層を別建て |
