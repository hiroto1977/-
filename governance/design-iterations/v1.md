# 12. 統合システム 設計図 (System Design)

> **目的**: 現存する全ての構成要素 (UI / AI / ストレージ / 監査 / ガバナンス / テスト / OS 別) を一つの設計に統合し、責務・信頼境界・不変条件・失敗モードを明確化する。
>
> **読み手**: AI 支援者・運用者・レビュア。
> **更新方針**: 反復的 (iteration) に書き直す。各反復で `design-iterations/v{N}.md` に履歴。
> **現バージョン**: v3 (本ファイル)

---

## 0. TL;DR

このシステムは **「ローカル ファースト × ガバナンス強制 × クロス OS 監査」** の三つを核に持つ業務 AI 運用基盤である。

- 全ての対話・ファイルは原則 **ローカル端末内** に閉じる (C0–C2)。クラウド AI を使う場合 C3 以上は **物理的に送信不能** (ローカル専用モード)。
- ルールは文書だけでなく **実行可能なフック / スクリプト** で強制される (pre-commit / pii-scan / audit chain hash / trash-first cleanup)。
- 監査ログは **bash と PowerShell が同じ JSONL に追記** し、SHA-256 連鎖で改竄検知。ブラウザ側で SubtleCrypto 検証可。
- テストは **依存ゼロ** (bash + node + python3 のみ) で 80+ 件、全層を覆う。

## 1. 階層モデル (Layer Model)

```
┌──────────────────────────────────────────────────────────────┐
│ L7  Governance Docs (governance/00–11, funding/, templates/) │  人間の意思決定
├──────────────────────────────────────────────────────────────┤
│ L6  User Interface                                           │  人間の操作
│      desktop/  v19/ui/  cowork/                              │
├──────────────────────────────────────────────────────────────┤
│ L5  AI Provider Abstraction                                  │  ブラウザ → API
│      Ollama (local) | Anthropic | Google                     │
│      ローカル専用モードで Anthropic/Google を遮断              │
├──────────────────────────────────────────────────────────────┤
│ L4  Operational Scripts                                      │  日次運用
│      preflight / storage-* / funding-deadline / pii-scan     │
├──────────────────────────────────────────────────────────────┤
│ L3  Safety Enforcement                                       │  commit / 削除 等の節目
│      pre-commit hook | trash-first | localStorage 警告       │
├──────────────────────────────────────────────────────────────┤
│ L2  Audit (Cross-OS, Tamper-Evident)                         │  全層が書き込む
│      lib/audit.sh + win/lib/audit.ps1 → ~/.claude/audit.jsonl│
│      audit-verify.sh + v19 #audit ビューア                   │
├──────────────────────────────────────────────────────────────┤
│ L1  Test Harness (依存ゼロ)                                  │  全層を検証
│      tests/{lib,unit,js,ps} + smoke-test.sh                  │
└──────────────────────────────────────────────────────────────┘
```

**設計原則 (各層で守る)**:
- **静的のみ**: ビルドステップなし。`python3 -m http.server 8000` だけで動く
- **ローカル ファースト**: 既定で外部送信ゼロ。クラウドは明示的選択のみ
- **依存最小**: bash / python3 / node があれば動く。npm 不要
- **冪等**: 同じ操作を 2 回実行しても結果は同じ
- **trash-first**: 削除は必ず一段経由する (即 `rm` 禁止)
- **chain-of-custody**: 全ての script が監査ログに行を残す

## 2. 信頼境界 (Trust Boundaries)

```
[Browser localStorage]──┐
[Filesystem ~/]─────────┼──── 端末内 (C0-C4 全て可) ────────────┐
[~/.claude/audit.jsonl]─┘                                         │
                                                                  │
[localhost:11434 Ollama]──── 端末内ループバック (C0-C4) ──────────┤
                                                                  │
[api.anthropic.com]──┐                                            │
[generativelanguage]─┼── 公衆網経由クラウド (C0-C2 のみ許可) ──────┤  ← 信頼境界
[rclone crypt:remote]┘                                            │
                                                                  │
[GitHub remote]──── push 時のみ (PII スキャン後) ─────────────────┘
```

**境界を越えるデータは必ず**:
1. **データ分類** に照らして適合判定 (`02_DATA_CLASSIFICATION.md`)
2. **PII スキャン** を通過 (commit 時は強制、それ以外は手動)
3. **監査ログ** に記録 (script は audit_log を呼ぶ)
4. **ローカル専用モード** ON 時は L5 の Anthropic / Google が UI から消える

## 3. データフロー (Data Flow)

### 3.1 通常会話 (Ollama, ローカル)
```
User → v19/ui/dashboard.js
     → POST localhost:11434/api/chat (NDJSON)
     → ストリーム解析 → DOM 描画
     → state.chat.sessions[i] (localStorage)
     → audit_log "v19.chat.send" (browser-side; not yet wired)
```

### 3.2 クラウド AI (Anthropic, Google)
```
User → v19/ui/dashboard.js
     → ローカル専用モード判定 (settings.localOnly)
        - ON  → 強制 ollama (reconcileLocalOnly)
        - OFF → 選択された provider に進む
     → POST api.anthropic.com or generativelanguage (SSE)
     → ストリーム解析 → DOM
     → state.chat.sessions[i]
```

### 3.3 commit
```
User → git commit
     → .git/hooks/pre-commit (symlink → scripts/hooks/pre-commit)
     → bash scripts/pii-scan.sh --staged
        - HIT  → exit 1 で commit ブロック
        - clean → 既存 audit.jsonl の verify (警告レベル)
     → audit_log "pre_commit.pass" "staged=N"
     → 通常の commit が継続
```

### 3.4 ストレージ運用 (月次)
```
cron / 手動 → bash scripts/storage-orchestrator.sh --routine monthly
           → storage-health.sh (診断)
           → storage-cleanup.sh --apply --aggressive (trash-first)
           → storage-archive.sh --plan (rclone, クラス別ルーティング)
           → audit_rotate (古い audit.jsonl 行を削除、チェーン切断)
           → audit_log 各ステップ
```

### 3.5 監査ログの読み取り
```
User → http://127.0.0.1:8000/v19/ui/dashboard.html#audit
     → loadAuditFile (file picker / ~/.claude/audit.jsonl)
     → for each line: SubtleCrypto.digest('SHA-256', prev_hash + body)
     → renderAuditEvents (timeline + 改竄行ハイライト)
```

## 5. 失敗モード (Failure Modes)

何が壊れたら何が起きるか・どう検出するか・どう復旧するか。

| 故障 | 影響 | 検出 | 復旧 |
|------|------|------|------|
| Ollama 未起動 | v19 で送信時 fetch エラー | preflight.sh / 接続テストボタン | `OLLAMA_ORIGINS=* ollama serve` |
| OLLAMA_ORIGINS 未設定 | ブラウザから 403 | DevTools Console + preflight | systemd / launchd / 環境変数で設定 |
| Anthropic / Google API キー失効 | クラウド送信時 401 | UI トースト | UI で再入力 |
| `audit.jsonl` チェーン切断 | verify が exit 1 | `audit-verify.sh` / pre-commit の警告 | 切断行以降の意図的ローテーションなら ok。そうでなければ調査 |
| `audit.jsonl` 削除 | 履歴消失。既存検証は通る (空ファイル) | preflight / 監査ビューアで「0 行」 | バックアップ復元 (要事前準備) — **現状ギャップ** |
| pre-commit hook が PII を見落とす | 機密が push される | (検出機構なし) | secret scanning (GitHub 側) — **現状ギャップ** |
| storage-cleanup が誤って重要ファイルを退避 | データ消失感 | trash 一覧で確認 | `--restore` で元の場所に戻す |
| storage-archive 中に rclone がクラッシュ | 部分転送 | 転送ログ + verify 段階で再実行 | rclone copy は冪等なので再実行で続きから |
| ブラウザ localStorage が満杯 | 新規セッション保存失敗 | UI に警告 (TODO) | 設定 → 全データ書き出し → 個別セッション削除 |
| Service Worker の旧バージョンキャッシュ | 古い UI が出続ける | バージョンずれ | 設定 → 「アプリを更新」(SW unregister) |
| Vision モデル未取得で画像送信 | "model does not support images" | UI トースト | `ollama pull llama3.2-vision` |

**現状の ギャップ** (未対応):
- (a) `audit.jsonl` のバックアップ戦略がない (本体ストレージに依存)
- (b) PII スキャンの偽陰性に対する セカンドライン がない (GitHub の secret scanning に頼る形)
- (c) ブラウザ localStorage の容量警告 UI がない
- (d) rclone 中断後のリジューム手順が文書化されていない

## 6. 保証と制約 (Guarantees & Limitations)

### 6.1 保証する (Honest)
- ✓ ローカル専用モード ON のとき、UI から Anthropic / Google を選択する経路は存在しない
- ✓ pre-commit hook がインストール済なら、PII の commit は (`--no-verify` 以外で) 不可能
- ✓ trash-first cleanup により、一度の操作で恒久削除されることはない (既定 30 日猶予)
- ✓ 監査ログは追記改竄が `audit-verify.sh` で検出される
- ✓ XSS: 既知の入力 (`<script>`, `javascript:` link, インライン event) は `tests/js/test_md.mjs` で阻止確認

### 6.2 保証しない (Limitations)
- ✗ ブラウザのメモリ ダンプ / DevTools 経由の API キー漏洩は防げない (localStorage 自体は OS のセキュリティに依存)
- ✗ Ollama のモデル ファイル自体の改竄は検出しない (model hash 検証は未実装)
- ✗ クラウド AI 側のログ保持や学習除外 はベンダーの設定 / ポリシーに依存 (`04_VENDOR_REVIEW.md`)
- ✗ ローカル専用モードは "選択肢を消す" だけで、すでに送信済の API 履歴を消すわけではない
- ✗ Windows PowerShell スクリプトの実機テストは手動 (Linux サンドボックスでは構造のみチェック)
- ✗ pre-commit hook は `--no-verify` で迂回可能 (CLAUDE.md ルールで人間承認を要求するが物理強制ではない)
- ✗ システム全体のシングルサインオン / RBAC はない (個人 / 小規模事業者向け設計)

## 9. 拡張 ポイント (Where to extend)

新しい AI プロバイダを追加する場合:
1. `dashboard.js` の `PROVIDERS` に登録
2. `LOCAL_PROVIDER_IDS` に入れるかどうか判定 (ローカル専用モード)
3. `tests/js/test_providers.mjs` に SSE / NDJSON 解析テストを追加
4. `governance/04_VENDOR_REVIEW.md` で評価記載

新しい script を追加する場合:
1. `scripts/<name>.sh` に書く
2. 冒頭で `source lib/audit.sh; audit_log "<name>.start" ""` を呼ぶ
3. `tests/unit/test-<name>.sh` を作る (`assert.sh` を `source`)
4. `README.md` の「実行可能スクリプト」節に 1 行追加

新しいガバナンス文書を追加する場合:
1. `governance/<NN>_<TITLE>.md` (採番は連番)
2. `governance/README.md` のナビに追加
3. `CLAUDE.md` の「ファイル構成」に反映

## 10. 既知の課題 (Open Issues)

§5 から再掲 + 設計面の課題:

1. **audit.jsonl のバックアップ戦略がない** — ストレージ障害で消失するリスク。提案: 月次ルーティンで `audit.jsonl.bak.<ym>` を作る
2. **PII スキャンの偽陰性に対するセカンドラインがない** — 提案: GitHub Actions で gitleaks 等を併用 (CI なしの方針と矛盾するためトレードオフ)
3. **ブラウザ localStorage の容量警告 UI がない** — 提案: 設定画面で使用量表示
4. **rclone 中断後の再開手順が文書化されていない** — 提案: `governance/10_STORAGE_HYGIENE.md` に節追加
5. **bash↔PowerShell の audit.jsonl バイト互換が手動検証** — 提案: tests/integration/audit-cross-os.sh で同じ入力 → 同じ JSON 行を検証
6. **storage-orchestrator のテストがない** — 提案: dry-all モードで sub-step が呼ばれるか確認
7. **cowork CLI のテストがない** — 提案: `--help` の出力検証 + Ollama 不在時の フォールバック確認
8. **Service Worker のバージョン管理戦略が不明確** — 提案: SW URL に hash クエリを付ける、または明示的な「更新」ボタン
